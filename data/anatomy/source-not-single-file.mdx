title: 데모 소스를 공개하며, 전시 01을 절반 되짚었다
exhibit: 12
category: 콘텐츠 전략
question: >-
  전시 01에서 나는 데모를 정적 단일 파일로 만들겠다고 했다. 메인 앱과 얽히지 않게, 빌드도 라우트도 공유하지 않게. 그런데 데모가 12종까지 늘자 욕심이 생겼다 — 이 원본 프로젝트를 자산으로 보관하고 싶고, 가능하면 사람들이 GitHub에서 가져갈 수 있게 하고 싶다. 그러려면 단일 파일을 포기하고 Vite·React 프로젝트로 만들어야 한다. 과거의 내 결정을 뒤집는 것인가, 아니면 그 결정의 무엇이 핵심이었는지 다시 보는 것인가.
options:
  - key: A
    title: 단일 파일 고수
    tradeoff: 전시 01의 결정을 지켜 일관되지만, 소스를 공개 자산으로 보관·배포할 수 없다.
  - key: B
    title: Vite·React 프로젝트로 전환
    tradeoff: "원본 소스를 GitHub에 공개하고 가져갈 수 있게 되지만, '단일 파일' 원칙을 절반 포기하고 빌드 파이프라인이 생긴다."
decision:
  key: A
  label: 격리는 지키되, 단일 파일은 양보
  sub: 전시 01의 결정을 상황 변화 앞에서 일부 뒤집었다
result: >-
  demos/ 폴더(공개 메인 레포)에 소스를 두고 빌드 산출물만 public/showcases/로 복사 — 격리는 유지한 채 소스를 공개했다. 웹 플랫폼·스타일 연구는 바닐라(소스가 곧 배포물), AI는 Vite+React로 유형마다 프레임워크를 달리했다. 각 카드에 GitHub 소스 링크가 붙어, 12종 데모 전부 가져갈 수 있게 됐다.
aiLedger:
  claude: 기존 코드를 삭제하고 코드를 생성하였다.
  me: 품질이 떨어지고 다른 곳에서도 쉽게 볼 수 있는 데모 예제들 선별하였다.
---

단순한 숫자를 채우는 것보다, 하나라도 의미가 있고 확장 가능한 것을 골라서 유지 보수하는 것이 타당하다.
