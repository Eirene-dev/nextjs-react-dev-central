---
title: 검색을, 정적에서 런타임으로 바꿨다
exhibit: 21
category: 원칙·아키텍처
question: >-
  검색 범위를 에세이와 해부까지 넓히려 했다. 그런데 발행된 에세이는 DB에서 오고(force-dynamic), 빌드 시점엔 그 내용을 알 수 없다 — 정적 search.json을 만들 수가 없다. 빌드타임 인덱스라는 익숙한 방식을 고수할 것인가, 아니면 런타임 API로 갈아탈 것인가.
options:
  - key: A
    title: 빌드타임 정적 인덱스
    tradeoff: 익숙하고 빠르지만, DB에서 오는 발행 에세이를 빌드 시점에 알 수 없어 search.json에 못 담는다.
  - key: B
    title: 런타임 API
    tradeoff: DB 동적 콘텐츠까지 검색에 넣을 수 있지만, 빌드타임 인덱스라는 익숙한 방식을 버려야 한다.
decision:
  key: B
  label: 제약을 만나, 경로를 바꿈
  sub: 빌드타임 인덱스를 버리고 런타임 API로
result: >-
  런타임 API(app/api/search-index, no-store)로 갈아탔다 — 발행 에세이가 DB(force-dynamic)라 정적 search.json이 원천적으로 불가능했기 때문. 단 이 전환에도 골든 277(콘텐츠 컬렉션·라우트 매니페스트)은 완전 불변이었다. 핑크 kbar 모달을 커스텀 kbar로 교체해 한글·코랄·종류별 섹션으로 맞췄고, public/search.json은 vestigial로 보존.
aiLedger:
  claude: 여러 구현 가능성을 점검했다.
  me: 검색바가 기존 Archive로 되어 있어서 변경을 명함.
