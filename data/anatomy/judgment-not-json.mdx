---
title: 해부를 JSON으로 쓰지 않았다
exhibit: 7
category: 콘텐츠 전략
question: >-
  해부 전시를 계속 쓰려고 한다. 구조 필드(질문·선택지·결정)와 장문의 근거 산문이 한 몸인 콘텐츠다. 일곱 편째부터 어떤 형식으로 작성하겠는가?
options:
  - key: A
    title: MDX + 템플릿
    tradeoff: frontmatter와 본문이 자연스럽게 한 파일에 담긴다. 대신 YAML 들여쓰기를 손으로 만진다.
  - key: B
    title: JSON 작성
    tradeoff: 구조 필드는 깔끔해진다. 대신 장문의 근거를 JSON 문자열 안에 쓰는 건 이스케이프·줄바꿈 때문에 고통스럽다.
  - key: C
    title: 웹 폼 + DB
    tradeoff: 어디서든 작성·즉시 게시. 대신 SSG를 잃고 콘텐츠가 git 이력 밖으로 나간다.
  - key: D
    title: MDX 저장 + 웹 컴포저
    tradeoff: 저장은 MDX(git) 그대로 두고 작성 경험만 폼으로. 대신 컴포저라는 도구를 따로 만들어야 한다.
decision:
  key: D
  label: MDX 저장 + 웹 컴포저
result: >-
  이 전과 다르게 JSON을 그대로 입력 후 저장하는게 아니라, 다시 또 내용을 편집하고 추가하는 방향으로 갔다. 그래야 진정한 내 판단의 기록이 되기 때문이다.
aiLedger:
  claude: 네 형식의 트레이드오프를 정리하고 컴포저 설계를 제안했다.
  me: 콘텐츠의 모양과 git 이력이라는 두 기준으로 D를 골랐다.
---

해부는 두 개의 다른 것이 한 몸인 콘텐츠다. 질문·선택지·결정 같은 구조 필드와, 판단을 풀어내는 장문의 근거. MDX는 frontmatter와 body로 정확히 이 둘을 나눠 담는다. 반대로 근거를 JSON 문자열 안에 욱여넣으면 이스케이프와 줄바꿈에 산문이 부서진다. 형식은 콘텐츠의 모양을 따라가야 한다.

더 큰 이유는 git이었다. 해부는 결정의 기록이다. 결정을 다루는 코너가 정작 자기 결정의 이력을 남기지 않는다면 앞뒤가 맞지 않는다. 즉시 게시되는 DB가 아니라, 검증을 통과해야 머지되고 머지가 곧 게시인 흐름 — 그 절차 자체가 이 코너가 하려는 말과 같은 말을 한다. C가 편한 줄 알면서도 고르지 않은 이유다.

그렇다고 YAML을 손으로 만지는 마찰까지 떠안을 필요는 없었다. 그래서 저장은 MDX 그대로 두고, 작성 경험만 폼으로 가져왔다. 도구가 형식을 대신 조립해 주되, 결과물은 여전히 git 안의 텍스트 파일이다. 그리고 이 전시 자체가 그 컴포저로 쓰였다.
