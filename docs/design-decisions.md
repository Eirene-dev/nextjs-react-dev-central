## 1. 전략 피벗 — 책 중심 → Essays + Showcases 중심

**결정**: 사이트의 무게중심을 **개인 에세이 + 쇼케이스**로 둔다. Book(judgment-dev 포함)은 한 섹션으로 후순위. 디자인도 책과의 시각적 연결 의무가 없다.

**근거**: 원 브리프는 "출간 책의 thesis에 맞춘 책-중심" 방향이었으나, 진행 중 우선순위가 바뀌었다 — 에세이는 *AI가 아닌 본인의 생각*, 쇼케이스는 *최신 AI/웹 + 혁신적·세련된 UI/UX 데모*가 사이트의 핵심 가치라는 판단. 빌드 우선순위도 essays/showcases 우선, book 후순위로 재배치.

**대안**: 브리프대로 책-중심 + 네이비 박스로 책과 시각 연결. → 채택 안 함.

**결정**: 5개 후보 중 **05**(SaaS 제품 랜딩, hero에 인터랙티브 데모 위젯) 채택. 산출물 `homepage-05-final.html`.

**비교 / 근거**:

- 02 다크 기술 — 혁신적 AI/웹을 강하게 시그널.
- 03 매거진 — 개성·임팩트, 편집 톤.

- **05 AI 프로덕트(채택)** — 인터랙티브 데모를 전면에 둬 "만든 것을 보여주는" 쇼케이스-중심 정체성과 가장 부합. 가장 "제품/도구" 톤.

**조정(05 → final)**: lab.mindvest.ai 연결은 **보류**하고 모든 동선을 **Showcases로** 집중. 마지막 CTA 배너는 lab 유도 → "쇼케이스 둘러보기"로 교체, lab은 **링크 없이 "곧 공개"** 표기만(배너·푸터). hero 데모 위젯 응답도 lab 대신 쇼케이스로 유도.

## 3. 디자인 시스템 — 로고에서 추출

**결정**: 책 대신 **로고(`Logo_RNC.png`)에서 팔레트를 추출**해 새 시스템을 만든다. 로고는 그대로 사용.
- 팔레트: **slate-navy(잉크/구조) + coral-red(액센트/에너지) + warm cream(서피스)**. 다크모드 = 딥 네이비-차콜. 색은 절제하고, **혁신/세련됨은 모션·타입에서** 낸다.
- 타이포: 본문 **Pretendard**(한국어 가독성 — 비협상), 기술 라벨 **JetBrains Mono**, (선택) 세리프 1종.
- 시그니처: 로고의 **동심원(rings)을 동적 SVG 모티프**로 확장(05의 데모 위젯 글로우/halo, 다른 후보의 레이더·hero 링 등). 정밀 일러스트는 출시 후.
- 컴포넌트: **완전 새로**(골격 구조만 재사용).

**근거**: 책 스펙(색/서체/일러스트)이 정의돼 있지 않고(Q1~Q3 모두 없음), 책과의 연결도 불필요해짐. 로고는 확정 자산이고 navy/coral/cream은 세련·distinctive하며 브랜드 통일을 공짜로 준다.

**기존과의 차이**: 현재 primary=핑크 + Space Grotesk(라틴 전용, 한국어 시스템 폴백) + AdSense는 "튜토리얼 사이트" 흔적 → 제거. Pretendard 도입은 한국어 읽기 사이트의 가독성·완성도에 가장 큰 레버리지.

## 4. URL 전략 — 유지 + 라벨링 (이전 안 대체)

**결정**: 기존 `/blog/*`·`/docs/*` **URL을 그대로 유지**(301 없음). `/archive`는 물리적 이전이 아니라 **라벨링된 인덱스 페이지**(+ 개별 글 상단 "아카이브" 배너). 내비에서 blog/docs를 빼고 **Archive 항목으로만** 접근. 검색엔진은 기존 URL로 동일 접근, **새 방문자는 홈 → Archive를 통해서만** 레거시에 도달. 홈은 레거시 블로그 피드를 내리고 **완전 새로**.

**근거**: 실제 SEO 자산은 blog 71 + docs 187 = **258편**. 이를 `/archive/*`로 옮기면 301 258건 + middleware matcher 258개 확장이 필요한데, 1주 일정에 **고위험·무이득**(내용 동일, 위치만 이동). URL 유지 = SEO 위험 0, 리다이렉트 작업 0. "Archive"는 내비/라벨 차원에서 처리.

**대안**: 전체를 `/archive/*`로 물리 이전 + 301. → 위험 대비 이득 없어 비채택.

## 5. Book — `/levelup/book` URL 유지 (브리프 대체)

**결정**: Next.js 책은 기존 `/levelup/book` **URL을 유지**한다. 브리프의 "`/book/nextjs`로 마이그레이션"을 **대체**.
- `/book` 인덱스 = **정적 2카드**(`booksData`). Next.js 책 카드 → `/levelup/book`, judgment-dev → `/book/judgment-dev`.
- `/levelup/book`은 **리스타일 없이** 레거시 룩 유지 + 상단 **"← Book" 브레드크럼**만. `Levelup` doc type 유지, `Book` 신규 추가. `/levelup` → `/book` 301.
- `/book/judgment-dev` = **MDX + 커스텀 컴포넌트**(정오표/보너스가 출시 후에도 자주 바뀌므로 편집 용이성 우선).

**근거**: 기존 책 페이지가 이미 그 URL로 열려 있고 구글 인입이 있어, URL을 옮길 이유가 없다(그룹 4 원칙과 일관: *대량 URL·내용 불변 → 유지*). 책이 후순위가 됐으므로 리스타일에 노력을 쓰지 않는다.

## 6. 의존성 — "동결" → "업그레이드 우선"으로 수정

**원래 결정**: 개편 중 의존성 **동결**, Next/Contentlayer 마이그레이션은 출간 후 별도 트랙. (근거: 1주 마감 하 위험 회피, 개편이 페이지 추가형.)

**수정된 결정**: **Next 13→16 + Contentlayer 제거를 Phase 0(최우선)**으로. 단 **마감이 열려 있을 때**.

**왜 바뀌었나**:
1. 타임라인이 열린 정황(judgment-dev 4순위, 업그레이드 우선 검토) — "빨리 출시" 압력이 약해짐.
2. 새 코드(홈·essays·showcases·book)를 **대량으로** 만든다 → 구 스택 위에 만들면 재작업.
3. **Contentlayer를 완전히 걷어내려는** 의도가 명시됨.
4. 폐기된 Contentlayer 0.3.4는 **Next 16과 깔끔히 공존하기 어렵다** → Next 업그레이드가 사실상 파이프라인 교체를 동반.

**전제(중요)**: *책 출간 D-7이 여전히 하드 마감이면* 이 대규모 플랫폼 마이그레이션을 먼저 하는 건 위험 → 그땐 **원래 "동결 후 출시"가 안전**. 이 마감 상태 확인이 선행 게이트(G1).

**파이프라인 후보**: **Velite 권장**(Contentlayer 계열, Zod, 마이그레이션 기계적, 활발히 유지) / next-mdx-remote·`@next/mdx`(제어 큼, Vercel 백업) / contentlayer2(드롭인이나 Contentlayer 계열이라 "완전 제거" 의도와 불일치, 비권장).

### 6-1. Phase 0 실행 결정 (2026-06-04 승인, 실행 확정)

분석·계획 검토 후 아래로 확정하고 `chore/next16-velite` 브랜치에서 실행한다.

- **(G1) 마감 열림 확정.** 책은 출판사에 넘어가 출시까지 수 주 여유 → Phase 0를 지금 진행.
- **목표 = Next 16 + React 19.** 파이프라인 교체와 프레임워크 업그레이드를 **분리**한다.
  - **안전 바닥 = 단계 1 종료(Next 13.4 + Velite, 빌드 그린, Contentlayer 제거).** 단계 2(Next 16)가 막히면 이 바닥으로 되돌리고 보고. 이미 깨끗·배포 가능 상태.
  - "Next 15로 후퇴"는 ***Next-16 고유* 비호환일 때만** 의미. ***React-19 고유* 비호환**이면 15로도 못 피하므로 의존성을 해결한다. 해결 순서: **정식 상향 → C-2 벤더링 → overrides(최후수단)**. overrides로 실제 비호환을 프로덕션에 숨기지 않는다.
- **pliny:** 정식 상향 우선, 막히면 **실사용 조각만 최소 벤더링**(완전 제거).
- **Node:** 로컬·CI·Vercel을 **22 LTS**로 통일(`package.json` engines + Vercel Node 설정 동기화).
- **골든 URL 매니페스트 검증(수용 기준 기계화):**
  1. 커밋된 stale `.contentlayer`가 아니라 **현재 스택으로 1회 빌드한 산출물**에서 추출.
  2. **draft 필터링을 contentlayer와 동일 재현**(초안 노출/누락 방지).
  3. **search.json(kbar) 스키마를 기존과 동일 형태 유지.**
  4. diff 비교 항목 = **path/slug + tag-data.json(태그 목록) + 생성 정적 라우트 수**.

> 결정이 추가로 바뀌면(예: pliny 벤더링 채택, 15 착지 등) 본 문서에 후속 기록한다.
