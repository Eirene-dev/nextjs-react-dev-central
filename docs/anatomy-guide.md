# 해부(Anatomy) 가이드 — 결정을 기록하고 전시하는 법

> 이 문서는 프로젝트 독립적이다. RNC(reactnext-central.xyz)의 /showcases/anatomy 가 기준 구현이며,
> 다른 웹·앱 프로젝트에서도 같은 형식으로 결정을 수집·작성할 수 있다.
> 원본 콘텐츠 예시는 docs/anatomy-content.md, 페이지 시각 기준은 docs/anatomy-exhibit-mockup.html 참조.

## 1. 해부란

실제로 부딪힌 결정 하나를 — 질문, 있었던 선택지, 내 결정, 근거, 결과, AI 분업 — 의 구조로 전시하는 기록이다.

존재 이유: AI가 코드를 쓰는 시대에 재현 불가능한 재료는 넷뿐이다 — 운영의 시간, 실제 사용자, 본인의 결정 기록, 본인의 취향. 해부는 그중 '결정 기록'을 콘텐츠로 만든다.

자격 시험 한 줄: **"이 항목을 다른 사람이 AI 프롬프트 한 줄로 재현할 수 있는가?"** 재현 가능하면 전시 자격이 없다.

## 2. 한 편의 구조 (6요소)

| 요소 | 역할 | 작성 지침 |
|---|---|---|
| 질문 question | 독자를 결정의 자리에 세운다 | 배경 2~3문장 + "당신이라면 어떻게 하겠는가?" — 2인칭 현재형 |
| 선택지 options | 실제로 검토했던 갈림길 | 보통 3개(2~4). 각각 제목 + 트레이드오프 한 문장(장점과 대가를 같은 문장에) |
| 내 결정 decision | 한 줄 선언 | 조건이 있으면 sub로 ("단, ~") |
| 근거 (본문) | 유일한 자유 산문 — 저자의 목소리 | 무엇을 골랐는지보다 **무엇을 버렸는지**가 핵심. 2~3문단, 읽기 5분 이내 |
| 결과 result | 결정이 실제로 낳은 것 | 모르면 `[추기 예정: X]`로 비워 둔다 — 시간이 지나야 채워지는 빈칸 자체가 mock이 못 가진 재료다 |
| AI 분업 aiLedger | AI가 한 것 / 내가 결정한 것 | 정직이 형식이다. 여기서 숨기면 코너 전체가 무너진다 |

## 3. 수집 워크플로 — 결정의 순간에

해부의 원료는 코드가 아니라 **그 순간의 갈림길 기억**이다. 일주일만 지나도 '있었던 선택지들'은 기억에서 사라진다. 그래서 다른 프로젝트를 개발하다 진짜 갈림길을 만나면, 그 자리에서 5분 스켈레톤을 남긴다(완성된 문장 불필요):

```
## [날짜] [한 줄 제목]
Q: (무엇을 정해야 했나)
A/B/C: (있었던 선택지 — 키워드만)
→ 결정: (+ 조건)
왜: (한두 문장 — 버린 것 포함)
AI: (AI가 한 일 / 내가 한 일)
결과: (미정이면 비움)
```

- 각 프로젝트 repo의 `docs/decision-log.md`에 누적한다. 렌더러가 없어도 로그 자체가 그 프로젝트의 자산이다.
- 분기마다(또는 5건 쌓이면) 전시감을 선별해 4장 기준으로 승격하고, RNC 형식(7장 템플릿)으로 완문한다.

## 4. 전시 승격 기준 (체크리스트)

승격 전 다섯 가지를 통과해야 한다:

- [ ] 실제 선택지가 2개 이상 존재했다 (사후 재구성한 선택지는 그렇다고 명시한다)
- [ ] 무엇을 버렸는지 한 문장으로 말할 수 있다
- [ ] 결과를 관찰할 수 있다 (또는 `[추기 예정]`으로 추적 가능하다)
- [ ] 질문이 독자에게도 성립한다 (그 프로젝트를 몰라도 투표할 수 있다)
- [ ] 사실관계 검증 목록을 만들었다 (수치·문구·시점 — 커밋 전 저자가 확인)

## 5. 데이터 스키마 (canonical)

어떤 저장 형식(MDX·JSON·DB)이든 이 모양을 따른다:

```json
{
  "slug": "kebab-case-english",
  "exhibit": 7,
  "category": "아키텍처 | 데이터 설계 | 운영·수익 | 제품 설계 | 콘텐츠 전략 | 브랜드·디자인 | ...",
  "title": "선언형 한 문장 (~했다 / ~인가)",
  "question": "...",
  "options": [
    { "key": "A", "title": "...", "tradeoff": "..." },
    { "key": "B", "title": "...", "tradeoff": "..." },
    { "key": "C", "title": "...", "tradeoff": "..." }
  ],
  "decision": { "key": "C", "label": "...", "sub": "단, ... (선택)" },
  "rationale_md": "근거 본문 — 마크다운 문단 (RNC에서는 MDX body)",
  "result": "... [추기 예정: X]",
  "aiLedger": { "claude": "...", "me": "..." },
  "date": "게시 시점 기입 (선택)"
}
```

규칙:
- `decision.key`는 `options[].key` 중 하나여야 한다 (RNC는 Velite 빌드 타임에 검증, 불일치 시 빌드 실패).
- slug는 영문 kebab-case. exhibit 번호가 정렬 키.
- `[추기 예정: X]`는 원문에 그대로 보존하고, 렌더 시에만 todo 칩 "X 추기 예정"으로 표시한다.

## 6. RNC 기준 구현 (참조)

- 저장: `data/anatomy/<slug>.mdx` — frontmatter = 구조 필드, 본문 = 근거. Velite `allAnatomy` 컬렉션, search/tag-data 제외(골든 277 불변), 페이지는 SSG.
- 투표: `anatomy_vote(slug, option, voter_token, UNIQUE(slug, voter_token))` — 1회 불변, localStorage 토큰(`anatomy-voter-id`), 공개 블록은 투표/건너뛰기 후 노출.
- 추가 절차: MDX 작성 → 사실 검증(4장 체크리스트) → 커밋 → 빌드(스키마 검증) → 머지(= 게시).
- 집필 규칙: AI는 기록에서 골격(질문·선택지·경과)을 조립하고 교정할 수 있다. **근거 문단의 판단 서술은 저자가 쓴다.** 완성 후 원문 그대로 — 생성·윤색·요약 금지.

## 7. 부록 — MDX 템플릿

새 전시는 이 파일을 복사해 `data/anatomy/<slug>.mdx`로 시작한다:

```mdx
---
title: TODO 선언형 한 문장
exhibit: 7            # 전시 번호 (오름차순 정렬 키)
category: TODO        # 아키텍처 / 데이터 설계 / 운영·수익 / 제품 설계 / 콘텐츠 전략 / 브랜드·디자인 ...
question: >-
  TODO 배경 2~3문장. 당신이라면 어떻게 하겠는가?
options:
  - key: A
    title: TODO
    tradeoff: TODO 장점과 대가를 한 문장에.
  - key: B
    title: TODO
    tradeoff: TODO
  - key: C
    title: TODO
    tradeoff: TODO
decision:
  key: C              # options[].key 중 하나 — 빌드 타임 검증
  label: TODO
  # sub: 단, TODO     # 조건이 있으면 주석 해제
result: >-
  TODO 결정이 낳은 것. [추기 예정: X]
aiLedger:
  claude: TODO AI가 한 것.
  me: TODO 내가 결정한 것.
# date: 2026-06-11    # 게시 시점에 기입
---

TODO 근거 본문 — 저자의 목소리로. 무엇을 버렸는지가 핵심이다. 2~3문단.
```
